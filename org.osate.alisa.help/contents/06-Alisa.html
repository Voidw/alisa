Incremental Assurance Workspace

Assurance Plans and Tasks

An assurance plan the set of veriifcation activities that must be performed in order to make a case of assurance for a system. The plan identifies the system and possibly subsystems to be verified by identifying a collection of verification plans for each. The assurance plan also identifies systems that are assumed to have been verified separately.

For each each assurance plan a user can define a set of assurance tasks. Each assurance task identifies a subset of the verificaiton activities that are to be performed as part of the specified assurance task. The subset can be specified either by defining a filter on requirement categories, verification categories, and selection categories, or the subset can be defined by identifying a set of claims of interest in the assurance plan. The filter is applied to the assurance plan to generate an assurance result instance that is used to track and report the assurance results.

<AlisaWorkArea> ::=
    alisa workareaname
    ( <AssurancePlan> | <AssuranceTask> )*

<AssurancePlan>::=
    assurance plan name=ID ( : "short descriptive title")?
    for <ComponentImplementationReferene>
    [
        ( description Description)?
        assert <VerificationPlanReference>+
        assume <AssurancePlanReference>+
       (issues ("explanation")+ )? 
    ]

<AssuranceTask>::=
    assurance task name=ID ( : "short descriptive title")?
    of <AssurancePlanReference>
    [
        ( description Description)?
        ( filter 
        ( ( requirements <RequirementCategory>+ )?
          ( verifications <VerificationCategory>+ )?
          ( selections <SelectionCategory>+ )?
        ) 
        | 
          claims <ClaimReference>+ ) 
        ( issues ("explanation")+ )? 
    ]

Uncertainty in Requirements and Arguments

Note: The concept of uncertainty has been taken from the draft RDAL Annex.

Uncertainty can be associated with requiremetns and with claims. In the case of requirements the uncertainty represents uncertainty the change of the requirement. This allows users to priortize requirments. In the case of claims it represents the uncertainty in the argument.

uncertainty [
volatility <number>
costImpact <number>
scheduleImpact <number>
familiarity <number>
riskIndex <number>
proportionalRiskIndex <number>
maturityIndex <number>
]

Number can be an real or integer.

  • volatility (mandatory): A ratio attribute representing the engineer's best judgment as to the probability the contractual element will change throughout the course of the project.
  • costImpact (mandatory): A positive factor attribute representing an impact factor on the development cost if this contractual element changes. The factor multiplied by the original cost shall yield the new cost.
  • scheduleImpact (mandatory): A positive factor attribute representing an impact factor on the development schedule if this contractual element changes. The factor multiplied by the original development time duration shall yield the new duration.
  • familiarity (mandatory): A ratio attribute representing the experience in providing solution for this contractual element. Experience has shown that a brand new element (familiarity = 0.0) will undergo a lot of evolutions and this must be taken into account.
  • timeCriticality (mandatory): A ratio attribute representing the time criticality, or priority with respect to time of implementing the parent contractual element. A high value means that the parent contractual element should be implemented in the early phases of the development process, as opposed to items not required until later phases of the project. For example, it will be important to address requirements having a high time criticality first.
  • riskIndex: This property can be used to store a risk index which may be computed from other properties of the contractual element.
  • propRiskIndex: This property can be used to store a proportional risk index computed from other properties of the contractual element.
  • maturityIndex: This property can be used to store a maturity index computed from other properties of the contractual element.

In addition, verification methods used in verification activities may have different degrees of effectiveness. This can be reflected in a weight factor associated the verification method.
